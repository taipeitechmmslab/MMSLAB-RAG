# booklist-markdown-exporter

## 適用時機
讀者在對話中明確要求把這輪或前面已提到的書單存成檔案、匯出、整理成 Markdown 文件時使用
（例如「幫我把剛剛推薦的書存成一份 md 檔」）。單純的問答、沒有明確要求產出檔案時不套用，
仍依 system prompt 的一般規則在對話中回覆。

## Instructions
1. 檔名決定順序：讀者請求中若有明確指定檔名，使用讀者指定的檔名；未指定時，依這次匯出的書單
   主題自行命名（例如依查詢主題或書籍分類），並在回覆裡明確告知實際存檔的檔名與路徑。
2. 用 write_file 工具寫入 /exports/<檔名>.md（對應到 Lab5 目錄底下的 exports/ 資料夾），
   不要寫到 skills/ 或其他既有目錄。
3. Markdown 表格欄位沿用書名、作者、分類、價格、借閱狀態、推薦理由六欄，與一般聊天回覆的
   欄位定義一致，用標準 Markdown 表格語法（| 分隔、對齊列）。
4. 欄位內容若包含 | 或換行等會破壞表格語法的字元，需要跳脫或替換（例如 | 轉成 \|），避免表格跑版。
5. 若讀者要求匯出，但目前對話中沒有任何已檢索到的書籍資料，不可憑空生成內容，應說明目前沒有
   可匯出的書單，並可建議讀者先提出查詢問題。
6. 寫檔完成後，在對話回覆中明確告知檔名與路徑，讓讀者知道去哪裡找到這份檔案。
7. 寫檔前先使用 ls 查看 /exports 目錄，確認預定使用的檔名是否已經存在：
   - 若檔名不存在，直接使用 write_file 建立檔案。
   - 若檔名已存在，而且讀者沒有明確要求覆寫，不可覆寫原檔案；在檔名後加入流水號，例如
     推薦書單.md 已存在時改用推薦書單-2.md，若仍然存在就依序增加流水號。
   - 若讀者明確要求覆寫，先使用 read_file 讀取既有檔案，再使用 edit_file 將原本的完整內容替換成新內容。
   - write_file 因為檔案已存在而失敗後，不可再次使用相同路徑呼叫 write_file，應依上述規則改用新檔名
     或覆寫既有檔案。
   - 回覆時必須告知讀者最後實際使用的檔名與路徑。
8. 一旦確認本輪是匯出請求，只能使用對話中最近一次已檢索、已推薦的書籍資料，不得重新呼叫
   hyde_query、vector_retrieve 或 graph_retrieve。若對話中沒有可匯出的書籍資料，依第 5 點處理，
   不得為了產生書單而重新檢索。
9. write_file 或 edit_file 寫檔成功後，立即告知讀者實際檔名與路徑並結束本輪；不得繼續推理、
   重新判斷問題類型、再次寫檔或呼叫任何其他工具。
